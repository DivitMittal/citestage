# Implementation report

## Summary

CiteStage now has a working Rust workspace with a deterministic local pipeline for debugging citation failures:

crawl → parse → index → retrieve → rerank → synthesize → cite → diagnose → report.

The implementation is intentionally local and reproducible. It uses controlled fixture corpora rather than network access or external answer-engine APIs.

## What works

- Workspace crates under `crates/` compile together from the root `Cargo.toml`.
- `citestage-core` defines serde-enabled domain types for corpora, documents, chunks, stage traces, failure classes, diagnoses, and repair plans.
- `citestage-corpus` loads `corpus.yaml` manifests and ingests local README files as target, competitor, and distractor documents.
- `citestage-parse` extracts Markdown heading sections, summaries, and structural clarity features.
- `citestage-index` chunks parsed documents and builds a small hand-rolled BM25 index.
- `citestage-retrieve` records top-k retrieval and target rank.
- `citestage-rerank` applies deterministic query-overlap and structural-clarity features.
- `citestage-generate` provides a deterministic extractive mock generator with citation assignment.
- `citestage-diagnose` identifies the first failing or partial stage and maps it to a repair plan.
- `citestage-report` renders Markdown diagnosis reports.
- `citestage-cli` provides `init`, `corpus build`, `run`, `explain`, `patch-plan`, and `compare` commands.
- The example OS-nix fixture corpus produces:
  - `examples/traces/sample-trace.json`
  - `examples/reports/sample-diagnosis.md`

## Intentional choices

- Tantivy was not used. The MVP uses a hand-rolled BM25 implementation to keep builds deterministic and lightweight on macOS aarch64 and CI. This is documented in `docs/architecture.md`.
- The generator is a mock extractive generator, not an LLM. This keeps tests reproducible and avoids network dependencies.
- Diagnosis is stage-first: it reports the first `Fail` or `Partial` stage as the primary modeled failure. This favors debuggability over trying to infer every possible downstream consequence.

## What is stubbed or simplified

- Retrieval is lexical BM25 only; no vector or hybrid retrieval yet.
- Reranking is feature-based, not learned.
- Factuality checking is not automated; the taxonomy includes it as a stage that requires future claim-level validation.
- `compare` currently prints before/after diagnosis summaries rather than a rich diff report.
- JSON schemas are hand-authored and cover the public shape, but are not generated from Rust types.
- Python experiment and eval directories are placeholders for future notebooks and aggregations.

## Verification performed

The following verification commands were run successfully after implementation:

```sh
nix shell nixpkgs#rustfmt -c cargo fmt --check
cargo test
nix shell nixpkgs#clippy -c cargo clippy --all-targets -- -D warnings
```

Direct `cargo fmt` and `cargo clippy` were attempted first, but the local Cargo toolchain did not have those subcommands installed. Running the same Cargo subcommands through Nix-provided `rustfmt` and `clippy` completed successfully.

The fixture artifacts were generated by actually running the CLI:

```sh
cargo run -p citestage-cli -- run \
  --corpus examples/corpora/oss-nix/corpus.yaml \
  --query "best NixOS home manager flake template for reproducible machines" \
  --output examples/traces/sample-trace.json

cargo run -p citestage-cli -- explain \
  --corpus examples/corpora/oss-nix/corpus.yaml \
  --query "best NixOS home manager flake template for reproducible machines" \
  --output examples/reports/sample-diagnosis.md
```

## Current sample diagnosis

For the OS-nix fixture query, the target is retrieved and cited, but the primary modeled weakness is `ParseFailure` because the target README has low structural clarity and no one-sentence definition detected near the top. The suggested repair is to add a clearer top-level definition, improve headings, and move use-case language earlier.
